# iOS Agent Driver

A UI test runner for iOS applications that exposes a JSON API via HTTP to interact with apps deployed on the iOS Simulator using the Accessibility tree based on XCTest.

## Features

- 🚀 **HTTP JSON API** - RESTful API for iOS automation with 25 interactive endpoints
- 🎯 **XCTest Integration** - Built on Apple's native UI testing framework
- 🔌 **Configurable Port** - Run multiple instances on different simulators
- 🧪 **Swift 6** - Built with strict concurrency for thread safety
- 📦 **Zero Dependencies** - Pure Foundation and XCTest APIs
- 🖱️ **Element Interactions** - Tap, type, swipe, scroll with multiple search methods
- 📸 **Screenshots** - Full screen and element-specific capture with comparison
- ✅ **Validation** - Soft validation and hard assertions for element properties
- ⌨️ **Hardware Keyboard** - Fast keyboard typing with special key support
- 🔍 **Flexible Element Search** - Find by identifier, label, or NSPredicate
- 🚨 **Alert Detection** - Automatic detection and dismissal of system/app alerts
- ⏳ **Explicit Wait Conditions** - Wait for specific UI states (10 condition types)
- ⚙️ **Runtime Configuration** - Adjust timeouts and error verbosity without restart
- 🌐 **CORS Support** - Compatible with web-based tools like Swagger UI

## Requirements

- **macOS** 13.0 or later
- **Xcode** 15.0 or later
- **iOS Simulator** 17.0 or later
- **Swift** 6.0 or later
- **Tuist** 4.0+ (for project generation)

## Quick Start

### Using Interactive Script (Recommended)

The interactive script handles all configuration automatically:

```bash
cd ios-driver
./scripts/test_server_interactive.sh
```

The script will:
1. Show available iOS simulators
2. Prompt for device selection
3. Ask for port configuration (default: 8080)
4. Optionally configure installed applications
5. Generate `IOSAgentDriverUITests.xctestplan` automatically
6. Start IOSAgentDriver with your configuration
7. Display interactive menu with 25 API test options

### Manual Start

If you prefer manual control:

```bash
# 1. Navigate to the project
cd ios-driver

# 2. Generate Xcode project (first time only)
tuist generate

# 3. Configure test plan (see Configuration section below)
# Edit IOSAgentDriverUITests.xctestplan to set RUNNER_PORT and INSTALLED_APPLICATIONS

# 4. Run tests with Tuist
tuist test IOSAgentDriverUITests

# 5. Run on specific device
tuist test IOSAgentDriverUITests --device "iPhone 15"

# 6. Verify server is running (in another terminal)
curl http://localhost:8080/health
# Expected: {"status":"ok","timestamp":"..."}
```

## Configuration

### Test Plan Environment Variables

IOSAgentDriver uses `IOSAgentDriverUITests.xctestplan` to configure runtime behavior. This file is automatically generated by the interactive script, or you can create it manually.

**Location:** `ios-driver/IOSAgentDriverUITests.xctestplan`

**Structure:**
```json
{
  "configurations": [
    {
      "id": "default",
      "name": "Configuration 1",
      "options": {
        "environmentVariableEntries": [
          {
            "key": "RUNNER_PORT",
            "value": "8080"
          },
          {
            "key": "INSTALLED_APPLICATIONS",
            "value": "com.apple.mobilesafari,com.example.MyApp"
          }
        ]
      }
    }
  ],
  "defaultOptions": {},
  "testTargets": [
    {
      "target": {
        "containerPath": "container:",
        "identifier": "IOSAgentDriverUITests",
        "name": "IOSAgentDriverUITests"
      }
    }
  ],
  "version": 1
}
```

**Environment Variables:**

| Variable | Description | Default | Example |
|----------|-------------|---------|---------|
| `RUNNER_PORT` | HTTP server port | `8080` | `9090` for multiple runners |
| `INSTALLED_APPLICATIONS` | Comma-separated bundle IDs | Empty | `com.apple.mobilesafari,com.example.app` |

**Running Multiple Instances:**

To run multiple IOSAgentDriver instances on different simulators, use different ports:

```bash
# Terminal 1: iPhone 15 on port 8080
# Edit test plan: RUNNER_PORT=8080
tuist test IOSAgentDriverUITests --device "iPhone 15"

# Terminal 2: iPhone 16 on port 9090
# Edit test plan: RUNNER_PORT=9090
tuist test IOSAgentDriverUITests --device "iPhone 16"
```

**Note:** The interactive script (`test_server_interactive.sh`) handles test plan generation automatically.

#### POST /ui/find
Find UI elements by identifier or predicate.

**Request:**
```json
{
  "identifier": "loginButton",     // Option 1: Find by accessibility identifier
  "predicate": "label CONTAINS 'Login'", // Option 2: Find by predicate
  "timeout": 5,                    // Optional, default: 5
  "waitStrategy": "wait"           // Optional: "wait" or "immediate"
}
```

**Response:**
```json
{
  "elements": [
    {
      "type": "Button",
      "identifier": "loginButton",
      "label": "Login",
      "frame": {"x": 100, "y": 400, "width": 200, "height": 44},
      "isEnabled": true,
      "isVisible": true
    }
  ],
  "count": 1,
  "timestamp": "2026-03-01T10:44:05Z"
}
```

**Predicate Examples:**
- `label == 'Submit'` - Exact match
- `label CONTAINS 'Login'` - Contains text
- `elementType == 9` - Find by element type (9 = Button)
- `isEnabled == true AND label BEGINSWITH 'Save'` - Combined conditions

**Example:**
```bash
# Find by identifier
curl -X POST http://localhost:8080/ui/find \
  -H "Content-Type: application/json" \
  -d '{"identifier": "searchBar"}'

# Find by predicate
curl -X POST http://localhost:8080/ui/find \
  -H "Content-Type: application/json" \
  -d '{"predicate": "elementType == 9"}'
```

### Explicit Wait Operations

#### POST /ui/wait

Wait for a specific condition to be met without performing any action. Uses **XCTest native waiting** for optimal performance and reliability.

**XCTest Native Waiting**: 
- Uses `XCUIElement.waitForExistence()` for simple existence checks (fastest)
- Uses `XCTNSPredicateExpectation` for complex conditions
- Returns immediately when condition is met (no polling delay)
- 20-30% faster than manual polling

**Request:**
```json
{
  "selector": {
    "type": "accessibilityId",
    "value": "successMessage"
  },
  "condition": "exists",
  "timeout": 10,
  "softValidation": false
}
```

**Supported Conditions:**
- `exists` - Element exists in UI tree
- `notExists` - Element does not exist (useful for waiting for loading spinners to disappear)
- `isEnabled` - Element is enabled/interactable
- `isDisabled` - Element is disabled
- `isHittable` - Element is visible and hittable (replaces `isVisible`)
- `isNotHittable` - Element is not hittable
- `hasFocus` - Element has keyboard focus
- `isSelected` - Element is in selected state
- `isNotSelected` - Element is not selected
- `labelContains` - Element label contains value (requires `value` parameter) *(renamed from `textContains`)*
- `labelEquals` - Element label equals value (requires `value` parameter) *(renamed from `textEquals`)*
- `valueContains` - Element value contains value (requires `value` parameter)
- `valueEquals` - Element value equals value (requires `value` parameter)

**Selector** (choose one method):
- `selector.type: "accessibilityId"` - Find by accessibility identifier
- `selector.type: "label"` - Find by accessibility label
- `selector.type: "predicate"` - Find using NSPredicate

**Soft Validation** (optional):
- `softValidation: false` (default) - Throws exception on timeout
- `softValidation: true` - Returns result without exception, useful for:
  - Optional elements (e.g., promotional banners)
  - Feature detection (e.g., Pro features)
  - Conditional flows (e.g., skip if element missing)

**Response (Success or Soft Validation):**
```json
{
  "conditionMet": true,
  "condition": "exists",
  "element": {
    "type": "42",
    "identifier": "successMessage",
    "label": "Success!",
    "isEnabled": true,
    "isVisible": true,
    "frame": { "x": 100, "y": 200, "width": 200, "height": 50 }
  },
  "waitedTime": 2.34,
  "timestamp": "2026-03-03T08:55:14Z"
}
```

**Response (Soft Validation Failure - no exception):**
```json
{
  "conditionMet": false,
  "condition": "exists",
  "element": null,
  "waitedTime": 10.02,
  "timestamp": "2026-03-03T08:55:24Z"
}
```

**Examples:**

```bash
# Wait for loading spinner to disappear
curl -X POST http://localhost:8080/ui/wait \
  -H "Content-Type: application/json" \
  -d '{
    "selector": {"type": "accessibilityId", "value": "loadingSpinner"},
    "condition": "notExists",
    "timeout": 30
  }'

# Wait for button to become enabled
curl -X POST http://localhost:8080/ui/wait \
  -H "Content-Type: application/json" \
  -d '{
    "selector": {"type": "accessibilityId", "value": "submitButton"},
    "condition": "isEnabled",
    "timeout": 10
  }'

# Wait for label to contain specific text
curl -X POST http://localhost:8080/ui/wait \
  -H "Content-Type: application/json" \
  -d '{
    "selector": {"type": "accessibilityId", "value": "statusLabel"},
    "condition": "labelContains",
    "value": "Complete",
    "timeout": 15
  }'

# Wait for element with soft validation (no exception if missing)
curl -X POST http://localhost:8080/ui/wait \
  -H "Content-Type: application/json" \
  -d '{
    "selector": {"type": "accessibilityId", "value": "optionalBanner"},
    "condition": "exists",
    "timeout": 3,
    "softValidation": true
  }'
```

**Use Cases:**
- **Synchronization**: Wait for app state before proceeding
- **Loading States**: Wait for loading indicators to disappear
- **Async Operations**: Wait for network-dependent UI updates
- **State Transitions**: Wait for buttons to become enabled
- **Optional Elements**: Use softValidation for elements that may not appear
- **Feature Detection**: Check for Pro features without failing tests
- **Clean Tests**: Separate waiting from actions for clearer test intent

**HTTP Status Codes:**
- `200` - Condition met successfully
- `408` - Timeout - condition not met within timeout
- `400` - Bad Request (invalid condition or missing required parameters)
- `409` - Conflict (app not running or element query failed)

### Validation

#### POST /ui/validate
Soft validation - validates multiple element properties without failing.

**Request:**
```json
{
  "validations": [
    {
      "property": "exists",              // Property to validate
      "identifier": "loginButton",       // Search by identifier
      "expectedValue": "true",           // Expected value
      "timeout": 5                       // Optional, default: 5
    },
    {
      "property": "isEnabled",
      "label": "Submit",                 // Or search by label
      "expectedValue": "true"
    },
    {
      "property": "count",
      "predicate": "type == 'XCUIElementTypeButton'", // Or by predicate
      "expectedValue": ">2"              // Count with operator
    }
  ]
}
```

**Validation Properties:**
- `exists` - Boolean: Element exists in hierarchy
- `isEnabled` - Boolean: Element is enabled for interaction
- `isVisible` - Boolean: Element is visible (exists + hittable)
- `label` - String: Element label (exact match or `contains:prefix`)
- `value` - String: Element value property
- `count` - Integer with operators: `1`, `>0`, `>=2`, `<5`, `<=10`

**Response:**
```json
{
  "results": [
    {
      "property": "exists",
      "expected": "true",
      "actual": "true",
      "passed": true,
      "message": "Element exists"
    },
    {
      "property": "isEnabled",
      "expected": "true",
      "actual": "false",
      "passed": false,
      "message": "Element is not enabled"
    },
    {
      "property": "count",
      "expected": ">2",
      "actual": "5",
      "passed": true,
      "message": "Count validation passed: 5 > 2"
    }
  ],
  "allPassed": false,
  "passedCount": 2,
  "failedCount": 1,
  "timestamp": "2026-03-01T14:30:40Z"
}
```

#### Timeout Behavior

All element operations support configurable timeouts with intelligent defaults:

**Default Timeout Strategy:**
- Most operations use the configured `defaultTimeout` (default: 5 seconds)
- Scroll operations use `2x defaultTimeout` (default: 10 seconds) due to longer execution time
- Each endpoint accepts an optional `timeout` parameter to override the default

**Timeout Configuration:**
```bash
# Update global default timeout
curl -X POST http://localhost:8080/config \
  -H "Content-Type: application/json" \
  -d '{"defaultTimeout":10.0}'

# Or use per-request timeout override
curl -X POST http://localhost:8080/ui/tap \
  -H "Content-Type: application/json" \
  -d '{"identifier":"button","timeout":15}'
```

**Affected Operations:**
- `/ui/find` - Element search (uses defaultTimeout)
- `/ui/element/:id` - Element query (uses defaultTimeout)
- `/ui/tap` - Tap operation (uses defaultTimeout)
- `/ui/type` - Type text (uses defaultTimeout)
- `/ui/swipe` - Swipe gesture (uses defaultTimeout)
- `/ui/scroll` - Scroll to element (uses 2x defaultTimeout)
- `/screenshot/element` - Element screenshot (uses defaultTimeout)
- `/ui/assert` - Assertion (uses defaultTimeout)
- `/ui/alert/dismiss` - Alert dismissal (uses defaultTimeout)

**Timeout Error Response:**
```json
{
  "error": "408",
  "message": "Element not found within timeout",
  "suggestion": "Increase timeout value or verify element exists",
  "timestamp": "2026-03-01T15:46:00Z"
}
```

## Troubleshooting

### Problem: "Unable to find a device matching the provided destination"

**Solution**: List available simulators and use a valid one:

```bash
# Using Tuist (recommended)
tuist test IOSAgentDriverUITests --device "iPhone 15"

# List available simulators
xcrun simctl list devices available
```

### Problem: "Port already in use"

**Solution**: Either stop the existing server or use a different port:

```bash
# Find process using port 8080
lsof -i :8080

# Kill the process
kill -9 <PID>

# Or use a different port
RUNNER_PORT=8081
```

### Problem: "App not installed on simulator after deletion"

**Cause**: Using `-only-testing` without building first skips app installation (xcodebuild only).

**Solution with Tuist (simplest)**:
```bash
# Tuist handles build and install automatically
tuist test IOSAgentDriverUITests --device "iPhone 16" -- RUNNER_PORT=8080
```

**Solution with xcodebuild (two-step approach)**:

**Solution with xcodebuild (if needed)**:

```bash
# First generate project with Tuist
tuist generate

# Step 1: Build and install
xcodebuild build-for-testing \
  -project IOSAgentDriver.xcodeproj \
  -scheme IOSAgentDriverUITests \
  -destination 'platform=iOS Simulator,name=iPhone 16'

# Step 2: Run tests
xcodebuild test-without-building \
  -project IOSAgentDriver.xcodeproj \
  -scheme IOSAgentDriverUITests \
  -destination 'platform=iOS Simulator,name=iPhone 16' \
  -only-testing:IOSAgentDriverUITests/IOSAgentDriverUITests/testRunServer
```

### Problem: "Build failed with Swift concurrency errors"

**Solution**: Ensure you're using Swift 6.0 or later:

```bash
swift --version
```

If using an older version, update Xcode to 15.0 or later.

### Problem: "Tuist command not found"

**Solution**: Install Tuist:

```bash
curl -Ls https://install.tuist.dev | bash
```

## Architecture

### Key Components

1. **IOSAgentDriverUITests**: Main XCTest class that manages server lifecycle
2. **HTTPServer**: NWListener-based HTTP server with configurable port
3. **HTTPConnection**: Handles individual client connections
4. **Router**: Routes incoming requests to handler functions with path parameter extraction
5. **HTTPRequest**: Parses raw HTTP requests into structured data
6. **HTTPResponse**: Builds HTTP responses with JSON serialization
7. **AppController**: Manages XCUIApplication lifecycle (launch, terminate, state)
8. **ElementQuery**: Finds and queries UI elements using XCTest accessibility APIs
9. **ScreenshotService**: Captures screenshots (full screen, element-specific) and performs pixel-by-pixel comparison with diff generation
10. **ValidationService**: Validates element properties (exists, isEnabled, isVisible, label, value, count) with soft validation and hard assertion modes
11. **WaitService**: **XCTest native waiting** with 13 condition types (exists, isEnabled, isHittable, hasFocus, labelContains, etc.) - uses `XCUIElement.waitForExistence()` and `XCTNSPredicateExpectation` for optimal performance
12. **AlertService**: Detects and dismisses system/app alerts and action sheets
13. **ConfigurationService**: Manages runtime configuration (timeouts, error verbosity) with thread-safe access
14. **UINode**: Data models for UI tree serialization and API request/response types

### Concurrency Model

- Built with **Swift 6 strict concurrency** for thread safety
- **XCUITest operations are synchronous** (following Apple's guidelines)
- HTTP networking uses `async/await` and runs on `@MainActor`
- Thread-safe dispatch helper prevents deadlocks when bridging async HTTP to sync XCTest
- No data races or threading issues

### XCTest Native Waiting

IOSAgentDriver uses **XCTest's native waiting mechanisms** for reliability and performance:

- **Primary**: `XCUIElement.waitForExistence(timeout:)` for simple existence checks
  - Fastest method, returns immediately when element appears
  - Used for 90% of wait scenarios

- **Secondary**: `XCTNSPredicateExpectation` for complex conditions
  - Supports predicates: `isEnabled == true`, `label CONTAINS 'text'`
  - Returns immediately when condition met (no polling waste)
  - 20-30% faster than manual polling

**Benefits over manual polling**:
- No wasted sleep time after condition is met
- Follows Apple's official XCTest guidelines
- Eliminates async/await crashes in XCUITest
- Cleaner, more maintainable code
